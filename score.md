我看了 `init.sql`，当前“评分”相关的数值和逻辑主要有 4 块：

## 1. 用户打分范围

`DrinkRecord.taste_score` 是用户对一次饮用记录的评分。

**允许范围：1 到 10 分。**

数据库里有两层校验：

```
taste_score INT CHECK (taste_score BETWEEN 1 AND 10)
```

以及触发器 / 存储过程里再次判断：

```
IF p_score < 1 OR p_score > 10 THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = 'Score must be between 1 and 10';
END IF;
```

也就是说，小于 1 或大于 10 都会报错。

------

## 2. 评分转化成用户偏好变化值 `v_delta`

在 `sp_add_drink_record` 里，评分会被转成一个偏好增减值：

| 用户评分  | `v_delta` | 含义                           |
| --------- | --------- | ------------------------------ |
| 9 - 10 分 | `+0.40`   | 非常喜欢，明显增加相关口味偏好 |
| 8 分      | `+0.30`   | 喜欢，增加相关口味偏好         |
| 6 - 7 分  | `+0.10`   | 还可以，小幅增加偏好           |
| 4 - 5 分  | `-0.05`   | 不太喜欢，理论上小幅降低       |
| 1 - 3 分  | `-0.20`   | 不喜欢，理论上明显降低         |

对应 SQL 是：

```
SET v_delta = CASE
    WHEN p_score >= 9 THEN 0.40
    WHEN p_score >= 8 THEN 0.30
    WHEN p_score >= 6 THEN 0.10
    WHEN p_score >= 4 THEN -0.05
    ELSE -0.20
END;
```

------

## 3. 用户偏好权重更新逻辑

每个咖啡都有一些口味标签，比如“奶香”“顺滑”“咖啡感”“浓郁”等。每个标签对这款咖啡有一个代表权重 `cft.weight`。

更新公式大致是：

```
本次偏好变化 = v_delta × 该咖啡标签权重
```

但是 SQL 里实际写的是：

```
GREATEST(0, v_delta * cft.weight)
```

这意味着：

**负数会被截断成 0。**

所以虽然前面设计了低分对应 `-0.05`、`-0.20`，但实际插入时低分不会降低用户偏好，只是不会增加。

如果用户已经有这个标签偏好，则更新：

```
weight = LEAST(5.00, GREATEST(0.00, weight + VALUES(weight)))
```

也就是：

```
新偏好权重 = 原权重 + 本次变化值
```

并限制在：

```
最小 0.00，最大 5.00
```

所以目前实际效果是：

| 评分   | 实际对偏好影响         |
| ------ | ---------------------- |
| 9 - 10 | 增加 `0.40 × 标签权重` |
| 8      | 增加 `0.30 × 标签权重` |
| 6 - 7  | 增加 `0.10 × 标签权重` |
| 4 - 5  | 不增加，也不减少       |
| 1 - 3  | 不增加，也不减少       |

这里有一个潜在问题：代码注释写的是“高分增加权重，低分降低权重”，但由于用了 `GREATEST(0, v_delta * cft.weight)`，低分并不会真的扣分。

------

## 4. 推荐分 `recommend_score` 计算逻辑

推荐咖啡时用的是 `sp_recommend_coffee`。

公式是：

```
recommend_score =
用户口味匹配分
+ 咖啡流行度 × 0.05
+ 该咖啡平均评分 × 0.10
```

SQL 是：

```
ROUND(
    COALESCE(SUM(up.weight * cft.weight), 0)
    + c.popularity * 0.05
    + COALESCE(AVG(dr_all.taste_score), 0) * 0.10,
    2
) AS recommend_score
```

具体解释：

| 部分           | 公式                          | 作用                                   |
| -------------- | ----------------------------- | -------------------------------------- |
| 用户口味匹配分 | `SUM(up.weight * cft.weight)` | 用户偏好的标签和咖啡标签越匹配，分越高 |
| 流行度加成     | `popularity × 0.05`           | 越多人/越高分喝过，推荐分略高          |
| 平均评分加成   | `AVG(taste_score) × 0.10`     | 历史平均评分越高，推荐分略高           |
| 保留小数       | `ROUND(..., 2)`               | 推荐分保留 2 位小数                    |

另外，推荐时会排除用户已经喝过的咖啡：

```
WHERE c.coffee_id NOT IN (
    SELECT coffee_id
    FROM DrinkRecord
    WHERE user_id = p_user_id
)
```

最后按：

```
recommend_score DESC, popularity DESC
```

排序，取前 10 个。

------

## 5. 流行度 `popularity` 的更新逻辑

每新增一条饮用记录，触发器会把这次评分加到咖啡的 `popularity` 上：

```
SET popularity = popularity + NEW.taste_score
```

所以：

```
咖啡 popularity = 所有用户历史评分总和
```

不是“饮用次数”，而是“评分累加值”。

例如：

| 饮用评分 | popularity 增加 |
| -------- | --------------- |
| 10       | +10             |
| 8        | +8              |
| 5        | +5              |
| 1        | +1              |

------

## 6. 咖啡标签权重的具体数值

`CoffeeFlavorTag.weight` 默认是：

```
1.00
```

部分核心标签会提高到：

```
1.20 或 1.30
```

目前出现的标签权重主要是：

| 权重   | 含义                        |
| ------ | --------------------------- |
| `1.00` | 普通相关标签                |
| `1.20` | 较重要/较能代表该咖啡的标签 |
| `1.30` | 非常核心的代表标签          |

例如：

| 咖啡类型/标签                    | 权重示例 |
| -------------------------------- | -------- |
| 美式的“咖啡感”“浓郁”             | `1.20`   |
| 拿铁的“奶香”“顺滑”               | `1.20`   |
| 馥芮白的“奶香”“顺滑”             | `1.30`   |
| 生椰拿铁的“椰香”“奶香”           | `1.30`   |
| 厚乳拿铁的“奶香”“浓郁”           | `1.30`   |
| 深烘相关的“深烘”“浓郁”“高咖啡因” | `1.30`   |
| 其他普通标签                     | `1.00`   |

------

## 总结版

当前评分逻辑可以概括成：

```
用户打 1-10 分
→ 评分转成 v_delta
→ v_delta × 咖啡标签权重
→ 更新用户对相关口味标签的偏好
→ 后续推荐时用 用户偏好匹配 + 流行度 + 平均评分 算推荐分
```

核心数值是：

```
评分范围：1 - 10

9-10 分：+0.40
8 分：+0.30
6-7 分：+0.10
4-5 分：-0.05，但实际被截断为 0
1-3 分：-0.20，但实际被截断为 0

用户偏好权重范围：0.00 - 5.00

咖啡标签权重：1.00 / 1.20 / 1.30

推荐分：
用户口味匹配分
+ popularity × 0.05
+ 平均评分 × 0.10
```

目前最大的逻辑注意点是：**低分不会真正降低偏好**。如果你想实现“低分扣除偏好”，需要把 `GREATEST(0, v_delta * cft.weight)` 改掉。
